Arrow IPC Stream file supportÂ¶

DataFusion can now read Arrow IPC stream files (#18457). This expands interoperability with systems that emit Arrow streams directly, making it simpler to ingest Arrow-native data without conversion.

Example (TODO: confirm exact syntax for IPC stream format selection):

STORED AS ARROW
LOCATION 's3://bucket/events.arrow';

Related PRs: #18457

Faster CASE expression evaluationÂ¶

DataFusion 52 completes major work from the CASE performance epic (#18075). Lookup-table based evaluation avoids repeated expression evaluation and reduces branching overhead, accelerating common ETL patterns.

Example:

SELECT
  CASE
    WHEN status IN ('NEW', 'READY', 'STAGED') THEN 'PENDING'
    WHEN status IN ('DONE', 'COMPLETE') THEN 'FINISHED'
    ELSE 'OTHER'
  END AS status_bucket,
  count(*)
FROM jobs
GROUP BY 1;

Related PRs: #18183

Extensible SQL planning with relation planner extensionsÂ¶

DataFusion now supports relation planner extensions for custom SQL syntax and planning logic (#17824, #17843). This lets downstream projects inject their own planning behavior without forking the SQL planner, which is critical for dialect extensions and custom table references.

ListingTable object store usage improvementsÂ¶

ListingTable improvements continue to reduce object store I/O and planning latency for partitioned datasets (#17214). DataFusion now normalizes partition and flat listings, enables a memory-bound list-files cache by default, and makes the cache prefix-aware for partition pruning.

Statistics cache improvementsÂ¶

The statistics cache has been improved to make pruning and planning more reliable in repeated workloads (#19051). DataFusion now exposes a statistics_cache function and improves cache memory behavior for listing workflows, making it easier to diagnose cache contents and reduce repeated I/O.

Pushdown expression evaluation via PhysicalExprAdapterÂ¶

DataFusion now pushes down expression evaluation into TableProviders using the PhysicalExprAdapter, replacing the older SchemaAdapter approach (#14993, #16800). This enables richer pushdown (expressions and projections) and improves consistency between logical and physical planning.

Diagram:

  v

Related PRs: #18998, #19345

Hash join build-side pushdownÂ¶

DataFusion can now push down build-side hash tables from HashJoinExec into scans (#17171). When the build side is small, DataFusion converts the hash table to an IN list or hash lookup that can be evaluated during scans, reducing the join input size early.

Example:

SELECT *
FROM orders o
JOIN small_dim d
ON o.dim_id = d.id;

Sort pushdown to sourcesÂ¶

DataFusion now supports sort pushdown into data sources, allowing scans to return sorted data or leverage reversed row groups when possible (#10433, #19064). This reduces memory pressure and can eliminate explicit sort stages for partitioned or pre-sorted data.

Example:

SELECT *
FROM parquet_table
ORDER BY event_time DESC;

Related PRs: #19064

DELETE/UPDATE hooks in TableProviderÂ¶

TableProvider now includes DELETE and UPDATE hooks, with MemTable providing the first implementation (#19142). This is an important step toward fully featured DML support and enables downstream storage engines to plug in their own mutation logic.

Example:

DELETE FROM mem_table WHERE status = 'obsolete';

Related PRs: #19142

CoalesceBatchesExec removal and integrated batch coalescingÂ¶

DataFusion continues the work from the CoalesceBatchesExec epic (#18779). The standalone CoalesceBatchesExec operator existed to ensure batches were large enough for vectorized execution, and it was inserted after filter-like operators such as FilterExec, HashJoinExec, and RepartitionExec. However, it also blocked other optimizations (like pushing limits through joins) and made optimizer rules more complex. This release integrates coalescing into the operators themselves and relies on Arrow's coalesce kernels, reducing plan complexity while keeping batch sizes efficient.

Diagram:

Before:
  Scan -> CoalesceBatches -> Filter -> CoalesceBatches -> Join

After:
  Scan -> Filter (coalesce inline) -> Join (coalesce inline)

Related PRs: #18540, #18604, #18630, #18972, #19002, #19342, #19239 Thanks to Tim-53, Dandandan, jizezhang, and feniljain for implementing this feature.

About DataFusionÂ¶

Apache DataFusion is an extensible query engine, written in Rust, that uses Apache Arrow as its in-memory format. DataFusion is used by developers to create new, fast, data-centric systems such as databases, dataframe libraries, and machine learning and streaming applications. While DataFusion's primary design goal is to accelerate the creation of other data-centric systems, it provides a reasonable experience directly out of the box as a dataframe library, Python library, and command-line SQL tool.

How to Get InvolvedÂ¶

DataFusion is not a project built or driven by a single person, company, or foundation. Rather, our community of users and contributors works together to build a shared technology that none of us could have built alone.

If you are interested in joining us, we would love to have you. You can try out DataFusion on some of your own data and projects and let us know how it goes, contribute suggestions, documentation, bug reports, or a PR with documentation, tests, or code.
